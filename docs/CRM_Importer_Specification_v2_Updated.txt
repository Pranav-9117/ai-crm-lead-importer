This version corrects a critical scope/timeline mismatch (original plan assumed weeks of build time; deadline is ~2 days out), fixes a workflow ordering issue (backend was being called before user confirmation), and adds business rules from the assignment that were missing from v1.0: allowed enum values, multi-email/mobile handling, skip-record rule, and CSV-safe output escaping. 1. Project Goal
Build an AI-powered CSV Import System capable of importing CRM leads from any valid CSV format without requiring users to manually map columns. Instead of assuming predefined column names, the application leverages an LLM to intelligently understand the CSV structure and transform it into the GrowEasy CRM schema.
Given the actual submission window, this revision prioritizes a working, well-engineered core flow (upload → preview → confirm → AI extraction → results) over auxiliary infrastructure. Bonus features are explicitly separated and time-boxed.

3. Corrected System Workflow
The original workflow called the backend (S3 upload + Import Job creation) immediately on file upload — before the user confirms. This violates the assignment's explicit requirement that no backend/AI call happens until after Confirm. Corrected flow:
User uploads CSV → file is parsed and previewed entirely client-side (e.g. PapaParse in the browser). No network call yet.
Preview table renders with sticky headers, horizontal/vertical scroll. No AI processing occurs at this stage.
User clicks Confirm → only now does the frontend call the backend API, sending the parsed CSV rows.
Backend validates payload → splits rows into batches → sends each batch to GPT-4.1 Mini with the mapping prompt.
Backend validates AI output against schema/enum rules → separates successful vs skipped records.
Backend returns a single structured JSON response: imported records, skipped records (with reasons), totals.
Frontend renders the result table: successfully parsed records, skipped records, total imported, total skipped.
Optional (if time permits): persist the Import Job / Leads / Skipped records to Postgres and the original file to S3, purely as bonus depth — not required for the core flow to work.
4. AI Prompt Engineering — Rules That Must Be Explicit in the Prompt
v1.0 referred to these only as generic placeholders ('Allowed enums', 'Mapping instructions'). They must be written into the actual system prompt verbatim, since this is the primary evaluation criterion.
4.1 Allowed crm_status values (enum — reject anything else)
GOOD_LEAD_FOLLOW_UP
DID_NOT_CONNECT
BAD_LEAD
SALE_DONE
4.2 Allowed data_source values (enum — leave blank if no confident match)
leads_on_demand
meridian_tower
eden_park
varah_swamy
sarjapur_plots
4.3 Multi-value fields (missing from v1.0 Feature 4.2)
Multiple emails found in a row → use the first as email; append the rest into crm_note.
Multiple mobile numbers found in a row → use the first as mobile_without_country_code; append the rest into crm_note.
crm_note is also the catch-all for remarks, follow-up notes, and any useful info that doesn't map to a defined field.
4.4 Skip rule (must be enforced in validation, not just prompted)
A record is skipped if it has neither an email NOR a mobile number. All other records must be kept, even if partially incomplete.
Every skip must be logged with: row number, reason, and the original raw row — for the skipped-records table in the UI.
4.5 Date format
created_at must always be a value parseable by JavaScript's new Date(created_at).
4.6 CSV-safety of AI output (missing from v1.0)
Each returned record must remain expressible as a single CSV row — no unintended line breaks.
If a note or field legitimately needs a line break, it must be escaped (e.g. \n) rather than inserted as a literal newline, so any CSV export stays valid.
This applies directly to the 'Download Processed CSV' feature in Phase 5 — the export step must escape newlines/commas/quotes per CSV spec (RFC 4180), not just concatenate raw AI text.

Version 2.1 Update — Final Implementation Decisions
This update refines the MVP while keeping the project achievable within the submission timeline. The focus remains a polished end-to-end AI CSV importer with reliable AI processing.
Configurable AI Batch Size
Use environment variables (AI_BATCH_SIZE and MAX_BATCH_RETRIES) instead of hardcoded values.
AI Response Validation Layer
Validate every AI response before returning it: JSON parsing, schema validation, enum validation, date normalization, phone/email normalization, and skip-rule enforcement.
Retry Policy
Retry only transient failures (HTTP 429, HTTP 5xx, network timeouts). Maximum 3 retries using exponential backoff (2s, 4s, 8s). Retry only the failed batch.
Prompt Versioning
Store prompts as versioned files (e.g., prompt_v1.md) for easier iteration and testing.
Stretch Feature Priority
After MVP: (1) Progress indicator, (2) Retry failed batches, (3) Dark mode, (4) Virtualized table, (5) Unit tests, (6) PostgreSQL + Import History, (7) AWS S3 storage, (8) Docker.
